## 问题 1：Pipe 模式的权限确认

**现状**：`echo msg | claude --resume` + `CI=true` 是纯单向通道。Claude 想做高权限操作时被直接拒绝，没有确认通道。`关vscode` 只是折中。

**核心矛盾**：pipe 模式设计上就是"一次性进出"，stdin 写完就 EOF，没有第二个交互回合。

**可能的思路**：

1. **预设权限白名单** — 在 Agent 端跑 `claude` 时传 `--permission-mode` 或 `--allowedTools`，把已知安全的操作预设为允许。但这不够灵活——你无法预知所有场景。

2. **两阶段模式** — 不用纯 pipe。Agent 先跑一次 `claude --resume`（非 CI 模式），用 `--print` 获取 Claude 的回复和 permission requests，Agent 解析后把权限请求发回企微让你确认，确认后再跑第二次带 `--resume`。代价是从"一次性"变成"可能需要多轮"，延迟增加。

3. **`--dangerously-skip-permissions`** — 如果你信任自己的项目目录，可以传这个 flag，Claude 不再询问权限。风险是 Claude 可以改任何文件。

**建议**：方案 2 最实用——Agent 检测 Claude 输出中的 permission denied 信号，自动把"需要确认什么操作"推送到企微，你回复"允许"后 Agent 重新执行。这样保持了企微作为交互通道，同时解决了权限问题。

## 问题 2：`@bridge:ask` 双向通信

**现状**：`@bridge:notify` 是 A → B → 用户。`@bridge:ask` 需要 A → B → A → 最终结果给用户。

**简洁方案**：

在 Gateway 层实现一个简单的"请求-响应"匹配：

```
A 发消息给用户: "帮我查数据库用户表结构 @bridge:ask 数据库专家 查 users 表"
                                    ↓
Gateway 拦截 @bridge:ask，转发给 B（数据库专家会话）
                                    ↓
B 执行查询，输出结果
                                    ↓
Gateway 检测到 B 的输出"属于"刚才 A 的 ask 请求
                                    ↓
Gateway 把 B 的回复作为新消息注入 A 的 stdin
                                    ↓
A 继续处理，最终结果返回用户
```

技术上只需要 Gateway 加一个 **pending ask 表**（内存即可）：
- A 发 `@bridge:ask` 时，Gateway 记录 `{fromSession: A的claudeId, askId: uuid, status: "waiting"}`
- Gateway 执行 B 会话，B 的输出被标记为 `askId` 的 response
- Gateway 把 B 的结果通过 Agent API 注回 A 的会话（追加一条 user message，然后 `--resume` 跑 A）
- A 继续处理，最终输出返回用户

这个方案不需要改 Claude Code 本身，只在 Gateway 层做路由逻辑。

## 1. 集群协作：锁机制 vs Git 分支隔离

**结论：在 AI 集群早期，"认领文件 + 快速提交" 配合“独占锁”效率最高；但随着并发增加，分支隔离是必然。**

* **Git 分支隔离（强推荐）**：
* 当会话 A 认领任务时，Bridge 自动（或让 A 自己执行命令）切出一个新分支：`feature/task-xyz`。

* A 完成后，提交 PR/MR。由专门的**验收会话**（见第三点）进行代码审查和 Merge。

Agent 跑完 `claude --resume` 后，扫描 stdout 找权限拒绝的信号。问题是 Claude Code pipe 模式没有结构化的 permission denied 输出——只能靠文本匹配，比如：

- `"requires approval"`
- `"blocked by permissions"`
- `"permission denied"`
- 或者 Claude 在回复里说"我被拒绝了"

这本身就是不可靠的——文本格式可能随 Claude Code 版本变化。

### 第二步：把权限请求发到企微

Agent 检测到之后，在 `/api/run-claude` 的响应里加一个字段 `permissionRequest`，Gateway 收到后发企微：

```
⚠️ Claude 需要权限：编辑 ~/.claude/settings.json
回复"允许"或"拒绝"
```

### 第三步：用户确认 → 重跑

用户回复"允许"，Gateway 需要知道这条"允许"不是新消息，而是对权限请求的回应。这意味着 Gateway 要维护一个 **pending permission 表**：

```
{ chatId, sessionId, whatWasBlocked, originalMessage }
```

收到"允许"后，Gateway 查表匹配，然后重新调 Agent 执行——这次带 `--permission-mode bypassPermissions` 或指定 `--allowedTools`。

### 第四步：重跑

重跑意味着把原始消息**重新执行一遍**。问题是：
- 第一次执行已经消耗了 token，重跑再消耗一遍
- Claude 的回复可能不一样（非确定性）
- 如果第一次执行里有些副作用操作（如创建了文件），第二次可能冲突

---

## 核心困难

1. **检测不可靠**：纯文本匹配，Claude Code 版本的输出格式不受我们控制
2. **状态管理复杂**：Gateway 要维护 pending permission 表，处理超时、用户忽略、多次权限请求等边界情况
3. **重跑浪费**：整个 conversation 重新跑一遍，token 翻倍
4. **实际上可能很少触发**：你的项目目录下的操作 CI 模式基本不拦，只有编辑全局文件（`~/.claude/`、`~/.gitconfig` 等）才需要权限。这类操作在正常开发中占比很低

- 真要遇到，用户切回 VS Code 手动处理一次就行

## `@bridge:ask` 实现方案

### 消息流

```
用户发消息给 A
  → A 处理，输出中包含 @bridge:ask B 查users表
    → Gateway 拦截，解析出目标=B，消息=查users表
      → Gateway 调 B：stdin = "[from=<A_uuid>] 查users表"
        → B 处理完毕，stdout = "users表有3个字段..."
          → Gateway 把 B 的回复注入 A：stdin = "[from=<B_uuid>][reply] users表有3个字段..."
            → A 继续处理，最终回复给用户
```

### 改动范围

只改 Gateway 的 `bridgeRoute()`，Agent 不需要改。

**`@bridge:ask` 分支逻辑**：
1. 正则匹配 `@bridge:ask <目标会话名> <消息>`
2. 查目标会话的 UUID（从 DB 或项目会话列表）
3. 告诉用户：`🔗 A → B 处理中...`
4. `execClaude(B_uuid, "[from=<A_uuid>] <消息>")` — B 知道是谁发来的
5. B 的 stdout 作为 reply
6. `execClaude(A_uuid, "[from=<B_uuid>][reply] <B的回复>")` — A 拿到 B 的回复继续
7. A 的最终 stdout 发给用户

**和 `@bridge:notify` 的区别**：notify 只做 A→B→用户，ask 多做一步 B→A→用户。

### 不变的部分

- `[from]/[to]` 用 UUID，Gateway 纯路由
- 企微显示用 aiTitle
- pipe 模式不变
