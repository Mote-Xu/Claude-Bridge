## Claude Bridge — 项目总览

### 🎯 定位

这是一个**手机↔电脑 Claude Code 桥接系统**。你在手机上通过企业微信发消息，控制你 Windows 电脑上的 Claude Code，实现"手机是电脑的第二屏幕"。

### 🏗️ 架构（三层）

```
企业微信 App  →  mote-home (Gateway)  →  Windows (Agent)
  webhook        Node.js :8933           Express :9877
                                       → claude.cmd --resume
```

- **Gateway** (`gateway/`) — 部署在 mote-home 服务器上，接收企业微信 webhook，路由消息
- **Agent** (`agent/`) — 部署在你的 Windows 电脑上，本地执行 Claude Code CLI
- 两者通过 Tailscale 内网 (`100.80.205.79`) 的 HTTP JSON 通信

### ✅ 已有功能（相当完整）

| 功能 | 说明 |
|------|------|
| 企业微信消息接收/回复 | 群聊 + 私聊，webhook 加解密 |
| 电脑会话发现 | Agent 扫描 `~/.claude/projects/` 自动发现项目 |
| 多项目路由 | 支持 `@项目名` 接入、`切换` 项目、`退出` |
| 多会话管理 | `@会话名` 创建/续接，序号选择历史会话 |
| 会话预览 | `预览 N` 查看会话详情 |
| 隐藏/取消隐藏 | 手动隐藏 + 自动同步 VS Code 隐藏列表 |
| `claude --resume` | 跨消息续接上下文 |
| 电脑离线检测 | 任务排队 + 30 秒定时 drain |
| Agent 看门狗 | Task Scheduler 每 5 分钟检查，挂了自启 |
| 热重载 | `/api/reload` → Agent 退出 → 看门狗拉起 |

### ⚠️ 已知问题 & 设计权衡

1. **VS Code 窗口会被关** — 手机发消息时 Agent 主动 `taskkill code.exe`，因为手机操作时人不在电脑前。回电脑重开 VS Code 会自动恢复所有会话。

2. **手机创建的新会话 VS Code 不显示** — 因为是 pipe 模式（`echo msg | claude`），非 TUI 模式创建的会话 VS Code 不一定能识别。但可用 `--resume` 续接。

3. **Agent 做了一些 hack** — 比如手动往 `~/.claude/sessions/` 写索引文件让 `--resume` 能查到会话，直接读 VS Code 的 `state.vscdb` SQLite 文件提取隐藏列表。

### 📁 代码量

- **Gateway**: ~490 行（消息路由 + 群聊逻辑）
- **Agent**: ~457 行（HTTP API + Claude 执行）
- **DB 层**: ~159 行（SQLite，6 张表）
- **企业微信**: ~89 行（加解密 + 消息发送）
- **Agent HTTP 客户端**: ~73 行

总共约 **1300 行业务代码**，架构清晰，职责分明。

### 🔧 技术栈

- Node.js + Express（两端都是）
- better-sqlite3（Gateway 数据层）
- 企业微信自建应用 API
- Tailscale WireGuard 组网

### 我的评价

这是一个**设计得很务实的项目**。代码量不大但功能覆盖很全——多项目、多会话、离线排队、看门狗、热重载都做了。架构分层清晰：Gateway 只管消息路由，Agent 只做本地执行，没有越界。

几个亮点：
- 纯 HTTP 架构（去掉了之前的 SSH），消灭了转义地狱
- 离线排队 + 定时 drain 保证了消息不丢
- 看门狗 + 热重载让 Agent 维护成本很低

潜在改进方向：
- `config.js` 里企业微信配置是硬编码占位符，实际值应该在 `PRIVATE.md` 或 `.env` 里
- Agent 的 `taskkill code.exe` 比较激进，如果以后支持不关 VS Code 的模式会更好
- Agent 读 `state.vscdb` 是正

## 2026-07-09 00:24
🤖: Gateway 就是一个**消息中转站**，跑在你的家庭服务器 mote-home 上。

---

### 物理上

```
/home/mote/ → /mnt/data/clawd/gateway/
              ├── index.js    ← 主程序
              ├── db.js       ← SQLite 数据库
              ├── agent.js    ← HTTP 客户端（调 Windows Agent）
              ├── wecom.js    ← 企业微信加解密
              └── config.js   ← 配置
```

一个 Node.js 进程，systemd 管理：`clawd-gateway.service`，监听 `127.0.0.1:8933`。

---

### 本质上

Gateway 做的事就三句话：

1. **收消息** — 企业微信把消息 POST 到它这里（通过 Cloudflare Tunnel → Caddy 转发）
2. **找目标** — 查数据库：这个群绑定了哪个项目？这个 `@会话名` 对应哪个 session ID？
3. **转发** — HTTP 调 Windows Agent 的 `/api/run-claude`，然后把 Claude 的回复发回企业微信

**它自己一行 AI 代码都没有。** Claude Code 是唯一推理引擎，Gateway 只是个"信使"。

---

### 为什么需要一个独立的 Gateway？

你可能会问：为什么不直接从企业微信调 Windows Agent？为什么要中间多一层？

**因为你的 Windows 电脑没有公网 IP。** 它在内网（192.168.x.x），只有 Tailscale 虚拟内网可达。企业微信的 webhook 只能 POST 到公网 HTTPS URL。

所以需要一台有公网入口的机器（mote-home + Cloudflare Tunnel）来做这个"跳板"：

```
公网                        Tailscale 内网
企业微信 ──→ Gateway ──→ Windows Agent
          Cloudflare    100.80.205.79:9877
          Tunnel
```

mote-home 刚好同时满足两个条件：Cloudflare Tunnel 给了它公网 HTTPS，Tailscale 让它能访问你的 Windows 电脑。

---

### Gateway 的状态机

Gateway 维护的核心状态很简单：

```
群聊 chatId ──→ 项目 (project_name + project_path)
                  │
                  ├── 会话 A (claude_session_id: "f2e6f385...", status: active)
                  ├── 会话 B (session_name: "重构", status: idle)
                  └── 会话 C (status: ended)
```

全部存在 SQLite 里。Gateway 重启不丢状态。

---

### 一句话总结

> Gateway = 企业微信消息 ↔ Claude Code 会话之间的**无状态路由器 + 有状态会话表**。它懂"谁在哪个项目、哪个会话该接谁的消息"，但完全不懂消息内容是什么。

## 第一部分：验证问题回答（系统现状确认）

### 1. 完整路径与生命周期

当用户在群里发送 `@重构 把用户模块换成 TypeScript` 时，完整流转路径如下：

1. **企业微信 App** 触发 Webhook，将加密的 XML/JSON 消息推送到公网。
2. **Node.js Gateway (mote-home)** 接收请求，进行企微解密。根据群聊/私聊上下文，识别出当前关联的“项目文件夹”。
3. Gateway 解析出目标会话为 `@重构`，将其放入队列，并通过内部路由通过 HTTP 请求转发给 Windows Agent：`POST http://100.80.205.79:9877/run-claude`，携带参数为该会话的 `id` 和用户消息。
4. **Windows Agent** 接收请求：
* 执行 `taskkill /f /im code.exe` 强制关闭 VS Code（释放 Session Lock）。
* 启动一个子进程，执行 `claude.cmd --resume <id>`。
* 通过标准输入（stdin）将 `"把用户模块换成 TypeScript"` 喂给 Claude 进程。


5. **Claude Code (唯一 AI 层)** 读取输入，调用本地工具（如读写文件、运行 Linter/Compiler）执行重构，并在完成后将响应输出到标准输出（stdout）。
6. **Windows Agent** 捕获标准输出，通过 HTTP 响应返回给 Gateway。
7. **Gateway** 接收响应，进行企业微信格式打包与加密，调用企微 Bot API 发送。
8. **用户手机**收到重构完成的通知和结果。

### 2. 电脑关机与开机的边界情况

* **电脑关闭时**：手机发送消息后，Gateway 尝试请求 Windows Agent（100.80.205.79），由于连接超时或连接被拒绝（Connection Refused），Gateway 会捕获异常。消息会被阻塞在 Gateway 的任务队列中（或直接向企微返回错误提示“Agent 离线”）。
* **电脑重新开机后**：Agent 服务由系统（如通过启动项或 VBS 守护）自动拉起，重听 9877 端口。Gateway 的健康检查（`/health`）恢复，如果队列中有积压任务（且未超时），或者用户重新发送，Gateway 将能正常转发，并通过 `--resume` 无缝接回断开前的 Claude 会话。

### 3. 核心原则违反分析

违反了“单一职责”与“端到端黑盒”原则（或称“不越权做 AI 推理”）。
Bridge 的核心定位是**消息总线与执行管道**。如果在 Agent 里加入 OpenAI 调用，会导致：

1. **状态割裂**：这部分推理无法进入 `claude --resume` 的 Session 历史中，破坏了上下文的连续性。
2. **架构退化**：Agent 变成了另一个微型 AI Agent，模糊了系统边界，增加了 Windows 端的依赖和环境复杂度（如需要额外配置 API Key、处理并发请求等）。

### 4. 为什么要 Taskkill VS Code？不关会怎样？

* **原因**：Claude Code 在运行时会在本地（如 `~/.claude/projects/...`）建立基于文件或进程的 **Session Lock（会话锁）**。如果用户的电脑上正开着 VS Code 且启用了 Claude Code 插件（或在集成终端里跑着同一个会话），该 Session 会被持续锁住。
* **不关会遇到的问题**：Agent 尝试 `claude --resume <id>` 时会遭遇 **Session Locked 错误**，导致进程直接退出或无限期挂起等待，手机端会收到失败报错，无法实现“电脑手机无缝双向同步”。

### 5. VBS 守护 vs 计划任务（Task Scheduler）

* **VBS 方案优点**：
1. **高实时性**：进程挂掉后 5 秒内就能拉起，几乎无感。
2. **轻量、低侵入**：不需要 Windows 系统的管理员权限去写入 Task Scheduler，纯用户态脚本运行。


* **VBS 方案缺点**：
1. **缺乏系统级保障**：如果 VBS 脚本进程自己因为异常或者注销被杀了，就彻底失去了守护能力。
2. **可观测性差**：没有标准的日志输出

### 3. 文件锁机制（`LOCK`）—— 解决“别动我的代码”

AI 的速度极快，如果架构师和后端专家同时想去改 `package.json`，必然冲突。既然多会话在同一台 Windows 上运行，我们可以利用**本地文件锁**（File-based Locking）：

* **规则**：当一个会话要大规模修改某个目录（如 `src/backend/`）时，它必须在 `.claude-cluster/LOCK/` 下创建一个名为 `backend.lock` 的空文件，里面写上自己的名字和 Session ID。
* **释放**：干完活、Git Commit 提交后，删掉这个 lock 文件。
* **冲突预防**：其他会话如果想碰这个目录，读到有 lock 文件，就必须**终止当前任务并 Yield 挂起**，通过总线给自己定个“闹钟”或者把 Token 还给 Gateway。
