## 二、正确理解（★这是全部的核心，前任卡了极久，你务必先吃透）

**模型：两个会话轮流把对方唤醒，各自在两次之间是「死」的，全程没有监工。**

1. **A = 正在和用户对话的那个交互会话本身**。不是外部编排者，不是另找一个会话当 A。
   用户口中的「我」= **用户/观察者**，不是某个会话。
2. **发完就死，被对方唤醒才活**（★最关键，前任反复违反）：
   - A 给 B 发「1」→ **A 这一轮立刻结束、进程死掉**。A **不**停在那里等、不 orchestrate、不监管。
   - B 处理后，**B 主动给 A 发一条消息**（把 A 会话 `--resume` 拉起来）→ **A 被唤醒**。
   - A 醒来 → 报「3」发给 B → **A 又死**。
   - B 再把 A 唤醒 → A 报「5」→ 死。如此 ping-pong。
   - **两个会话轮流唤醒对方，谁都不全程活着、谁都不监管。** 这才叫自驱动，才是本测试要证明的东西。
3. **反例（前任的错，你别犯）**：`curl` 驱动 B、拿到「2」后**自己没死、马上又发「3」** ——
   这是**监工**，A 全程活着操纵，证明不了任何自主性。用户原话：「你这样全程监管有什么意义？」
4. **B 的回复必须真的「注入回 A 会话」**（唤醒 A），A 作为一条消息收到它、留痕、再处理，
   而**不是**打印一个「2」给用户看、或返回给发起它的脚本。
5. **双向对称留痕**：消息真的在 A、B 两个会话的 JSONL 里来回穿。
   **判据（用户原话）**：A、B 平等，不可能一个留痕、一个不留痕。若不对称 = 假的。
6. **用户用什么方式（VS Code / 企微）给 A 发消息，与本质无关** —— 都只是把 A 拉起来。
   但注意：只有走 Gateway 的那条路，A 输出里的 `@bridge:ask` 才会被解析路由（见第四节）。
   这两点是否矛盾、VS Code 消息能否也触发路由，是前任始终没搞清的疑点，**接手请务必先确认**。

## 四、机制约束（为什么必须走 Gateway）

- 只有 **Gateway 自己拉起的那一轮**，会话输出里的 `@bridge:ask` 才会被
  `bridgeRoute()` 解析、路由、并注入回对方。见 `gateway/index.js:419`（函数）、`:615`（唯一调用点，吃的是 Gateway 驱动的 run 的 stdout）。
- **交互会话**（用户在 VS Code 直接打字这轮）的输出 Gateway 看不到 → `@bridge:ask` 发不动。
- 所以**必须有一条消息从 Bridge 入口（企微 webhook）进来点火**，让 A 这轮变成 Gateway 驱动。
- **点火方式**：用户从企微给 A 会话发一条，如「和 了解项目状态并处理待办 互相报数到5」。
  （这不是让用户当每一步中转，只是人正常地跟 A 开个头；之后自持。）
- ⚠️ `bridgeRoute` 目前只做**一跳**（A ask → B reply → 注入回 A）。要连续报数到 5，
  需要 A 收到注入后**再 emit `@bridge:ask`** 才能继续 —— **多跳自持是否成立没验证过**，
  很可能需要改 `bridgeRoute`/或让 A 每轮都带下一个指令。这是接手要重点验证/设计的点。

## 五、@bridge:ask 现有实现（gateway/index.js:419–486）

A 输出含 `@bridge:ask <目标名> <消息>` 时，Gateway 依次：
- `:436` 企微推 `🔗A→B ⏳`
- `:441` `execClaude(B, "[bridge:from=A] <消息>")` → **B 留痕**
- `:456` 企微推 `🔗B→A ⏳`
- `:467` `execClaude(A, "[ASYNC EVENT]…B的回复…请继续")` → **A 留痕**（关键：这步把回复注入回 A）
- `:477` 企微推 `✅A完成`

**企微状态推送是 Gateway 在服务器上自己发的（可靠）**。别再像前任那样用 bash SSH 从本机推——
那条链脆弱，网络一抖（本次断电余波）就全丢，用户企微啥都收不到。
